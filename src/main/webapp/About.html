01 — Mission

Why we write.
Why it matters.

The internet is full of shallow tutorials that get you to "hello world" and stop there. ByteLog exists to go further — into the internals, the edge cases, the production failures, and the hard-won lessons that only come from actually building things.

Every article on ByteLog is written by a practising engineer who has run the code in production, hit the failure mode being described, and taken the time to write it down properly. No AI-generated filler. No sponsored content. No paywalls.

The ByteLog Auth module — the Java Servlet + JDBC + MySQL authentication system this site is built on — is itself a working example of everything we preach: clean separation of concerns, secure password hashing with bcrypt, and zero external framework dependencies.

// principle_01

Depth over clicks

We would rather publish one 15-minute deep dive per week than seven thin summaries. Quality compounds; quantity doesn't.

// principle_02

Code that compiles

Every snippet on this site has been compiled and run. If it's in a tutorial, it works — or we say exactly why it doesn't and what to do instead.

// principle_03

Security is not optional

From bcrypt cost factors to PreparedStatement usage, we treat security as a first-class concern — not an afterthought bolted on at the end.

// principle_04

No ads, ever

ByteLog has never and will never run display ads or accept sponsored posts. The only agenda is good engineering content.

02 — Tech Stack

What ByteLog
is built on.

☕

Java 17

Core language. LTS release with records, sealed classes, and pattern matching. Compiled to a WAR and deployed on Tomcat.

// JDK 17 LTS
⚙

Jakarta Servlets 6

Request handling with @WebServlet annotations. SignUpServlet, SignInServlet, DashboardServlet, SignOutServlet.

// jakarta.servlet 6.0
🐬

MySQL 8

Relational persistence with a users table and login_audit table. UTF8MB4 charset throughout.

// mysql-connector-j 8.3
🔑

jBCrypt

Password hashing with a configurable work factor (default cost = 12 ≈ 300ms). Constant-time verification prevents timing attacks.

// org.mindrot:jbcrypt:0.4
📄

JSP + JSTL

Server-side views with <c:out> for XSS-safe output. No client-side framework — just semantic HTML.

// jakarta.tags.core
📦

Maven 3.9

Build, dependency management, WAR packaging. Single command to compile, test, and deploy: mvn clean package.

// maven-war-plugin 3.4
🪵

SLF4J + Logback

Structured logging facade with a rolling-file appender. Password hashes are never logged. Sensitive params excluded from toString().

// logback-classic 1.5.3
🐈

Apache Tomcat 10.1

Jakarta EE 10 compliant servlet container. Session cookies set to HttpOnly + SameSite=Strict in web.xml.

// servlet container
🧪

JUnit 5

Parameterised unit tests for all validators and the password utility. Tests run without a database or servlet container — fast and isolated.

// junit-jupiter 5.10.2
03 — Architecture

Layered, deliberately
simple.

ByteLog Auth follows a classic three-tier architecture with no framework magic. Every class has a single job. Every SQL query uses a PreparedStatement. Every password is hashed before it touches the database.

View (JSP) signup.jsp · signin.jsp · dashboard.jsp ↓
Controller (Servlets) SignUp · SignIn · Dashboard · SignOut ↓
Utility Layer PasswordUtil · ValidationUtil · AppConfig ↓
Data Access (DAO) UserDAO + DBConnection ↓
Database (MySQL) users · login_audit
